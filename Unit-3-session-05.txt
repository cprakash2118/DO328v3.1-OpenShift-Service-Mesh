Step-1-# DestinationRule
$ oc explain ds |less
$ oc explain ds.spec |less
$ oc explain ds.spec.host |less

$ oc explain vs |less 
$ oc explain vs.spec |less 


vim destinationRule-file.yaml 
apiVersion: networking.istio.io/v1  
kind: DestinationRule
metadata:
  name: product-dr 
spec:
  host: product-service
  subsets:
  - name: v1 
    labels:
      version: v1 
  - name: v2
    labels:
      version: v2
	  
Step-2-# VirtualService

vim virtualService-file.yaml 
apiVersion: networking.istio.io/v1  
kind: VirtualService 
metadata:
  name: product-vs 
spec:
  hosts:
  - product-service
  http:
  - route:
    - destination:
	    host: product-service
	    subset: v1 

Outcome -> All traffic toward v1

Introduction to OpenShift Service Mesh Traffic Management (LAB):-
- Understand how traffic flows inside the service mesh (OSSM).
- How to control traffic using istio resources (OSSM based on istio project).
- Major 3 components:-
  1-) Gateway
  2-) VirtualService
  3-) DestinationRule
  
Application deployed on Kubernetes:-
 - Deployment -> ReplicaSet -> POD (Application)
 - service (svc) -> Communication with PODS (3 pods) - Internal only 
 - route will be created w.r.t service (Exposed application deployed on above pod)
 
 w.r.t normal k8s env:-
 - service -> Load balance traffic randomly (LB=RR).
 
 w.r.t Service Mesh env:-
 - You can control:-
   - Who gets traffic
   - How much traffic
   - Based on what condition.
   FYI- This  is done using "istio's CRDs"
   
What you build in this LAB:-
- Deploy applications (Sample app like showroom/bookinfo):-

Step-1:- Expose application via Gateway.
  FYI- Like OpenShift "route" but controlled by service mesh (OSSM).

Step-2:- Define traffic rules via VirtualService (oc get vs).
  Example:-
     What kind of controls?
     - Routing 
     - Matching rules 
     - Traffic split

Step-3:- Define policies via DestinationRule (oc get dr).
   Example:- 
      What kind of control managed here?
      - Versions (subsets)
      - Load Balacing 
      - Later TLS 

  End-to-End flow:-
  User (WB) -> Gateway (gw) -> VirtualService (vs) -> DestinationRule (dr) -> Pod (via Envoy).
