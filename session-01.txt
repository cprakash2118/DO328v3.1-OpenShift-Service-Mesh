- Red Hat OpenShift Service Mesh (Deafult with RHOCP) - Add-on service w.r.t Red Hat OpenShift.
- OpenShift Service-Mesh 2.1 vs 3.1 

Componets:-           Service-Mesh V2.1                       Service-Mesh V3.1 
1- Istio              ~1.10-1.2                               ~1.8+ (Major jump)
2- Envoy               Older LTS (Low perfomance issue)        Newer (Performance optimized)                               
3- Jaeger               Old                                    Moving to open telemetry part                                     
4- Kiali              v1.x                                     v2.x (new UI + insight)

Service-MeshV2.1 based on:-
- Heavy sidecar-based model 
- Every pod -> Envoy injected 
- CP (Control Plane - Istiod) managed all 

Problem Statement:-
- High resource usages (CPU/Memory)
- Complex debugging 
- Slow scaling in larger env (Big CLuster).

- Service Mesh conceps

Questions:-
- Why did service mesh become necessary in modern microservices platfrom ?
- Why OpenShift Service Mesh required w.r.t Enterprise env 

Problems:-
- Multiple services talking to each other.
- Hard to trace requests end-to-end 
- security b/w services is inconsistent 
- Retry/timeouts/Circuit breaking logic get duplicated in code.
- Monitoring part 
- Ownership part (App/Platform teams).

Solution:-
Service Mesh - OpenShift Service Mesh addresses these corss-cutting probelms in a platfrom-driven way.
### Building Resilient Microservices with Istio and Red Hat OpenShift Service Mesh:-
- Resilient Microservices:-
  - Resilient means service continue to operate reliably even when failure happen.
    
	Some failure list:-
	- One pod crashes 
	- Downstream service becomes slow.
	- Network latency 
	- Certifcate issue 
	- Traffic suddenly spikes 
	  etc 
	  
	Resilient means that platfrom can help with:-
	- retries 
	- timeouts 
	- Failover behavior 
	- traffic control 
	- LB 
	
  - Istio:-
     Istio is the service mesh technology foundation and it provides.
	 - Traffic Mgmt 
	 - Security 
	 - Policy control 
	 - Observability 
	 
   - Red Hat OpenShift Service Mesh (Istio + Envoy + Jaeger + Kiali )
   
# Red Hat OpenShift Service Mesh:-
- Service Mesh is an infrastructure layer that manages service-to-service communication in a microservices env.

# Why service mesh is needed in Microservices?
Monilithic apps -> Many internal calls happen inside one process.
Microservices   -> One user request my travel through:-
  - API gateway 
  - Frontend 
  - Product service 
  - cart service 
  - payment service 
  - recommendation service 

 Now think this way :-
 - Traffic complexity 
 - Security complexity
 - Observability complexity
 - Policy consistency 

# Core Service mesh Concepts:-
- Data Plane  Where actual application traffic flows.
- Control Plane  - Control plane like a brain which define policies, controls traffic behavior, manage certificate/identity
