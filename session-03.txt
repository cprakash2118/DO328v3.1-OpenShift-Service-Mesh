#- Service Mesh Showroom Application:-

point-01-# Bookinfo Application deployment

- Bookinfo is not just demo application, it is controlled microservice deployment.
- Bookinfo help us to understand:-
  - Traffic routing 
  - Observability 
  - Security Behaviour 
  All these without changing application code.
  
  - It also states that each microservice is deployed with an "Envoy sidecar", which gives the mesh the hooks needed for external control.
  
Bookinfo like a small online bookstore page (application).
- Once user access this Bookinfo application (When a user opens the page, they only see one webpage only).
- But in the backend, that page is build by multiple serivce working together.
Example:-
  Services:-
  Productpage -> Main frontend page
  details -> book metadata 
  review  -> User reviews 
  ratings -> Starts for ratings
  
  FYI
  - we will discuss/explain that these service are written in different languages and collaborate to render the final page.
  
  - Review service having multiple version to be deployed (Canary routing).
  
  
POint-02-# Why Red Hat uses Bookinfo for Service mesh [OSSM point of view how Bookinfo deployed].
- Because a service mesh is useful only when there is service-to-service communication complexity.
- Bookinfo gives exactly that:-

Why this application is perfect:-
1-# Multiple Services:-
    Once request touches several services. This way you can see complete traffic (East-west traffice) see clearly.
	
2-# Multiple Languages:- 
- Python, Ruby, Java, NodeJS are used, showing that service mesh works regardless of application's languages/framework. 
- OSSM provides a uniform layer regardless of application languages.

3-# Multiple Versions of one service:- 
- review has v1,v2,v3.
- This is the core reason traffic shifting is easy to demo.

4-# No code changes needed for traffic control.
- Traffic behavior is changed by OSSM resource like - VirtualService, DestinationRule etc not by modifying application logic.
- OSSM moves communication logic out of the application and into centralized policy/configuration.

5-# Observability become visible.
- Once Bookinfo application frontend page used by user/customer then multiple backend calls - kiali and tracing can how tology, traffic flow and dependencies.

6-# Security use cases fit here - Because service to service communication continuosusly, it is easy to show why mTLS matters and how reousres like "PeerAuthentication" can secure these communication b/w multiple microservice.

POint-05-# Where OSSM comes in:-
  Your app container says:- "I want to call reviews service".
  But in reality, traffic path become more like this flow:-
  APP container -> Local Envoy sidecar -> Destination Envoy sidecar -> Destination App.

  Your application thinks it is doing a normal service calls but "Envoy Proxy" is:-
  - Oberving 
  - routing 
  - securing 
  - collecting metrics 
  - propagating tracing context.
