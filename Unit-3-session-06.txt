Point-02-) Business behind this LAB.
"reviews" service has multiple versions:-
 - v1 -> reponse shows "No starts"
 - v2 -> reponse shows "Black starts"
 - v3 -> reponse shows "Red starts"

  These visible responses are used as an easy indicator of which subset received the request.
  gw -> vs -> dr then "vs" will route to pod.

Point-03-) End-to-End flow:-
 Client -> Istio ingress gateway -> VirtualService match rukes -> DestinationRule subset mapping -> correct reviews pod version   

   - VirtualService decides where traffic should go 
   - DestinationRule defines what v1,v2,v3 actually refered 
   - Envoy sidecars enforce the rules at runtime.   
   FYI- Without "DestinationRule", subset-based routing cannot work correctly.

Implemented header-based routing:- 
x-user-type: internal -> v3 -> red stars 
x-user-type: beta -> v2 -> Black stars
everyone else -> v1 -> No stars


Implement URI path-based routing:-

path rules.
/api/v3/reviews/* -> v3 -> Red stars
/api/v2/reviews/* -> v2 -> Black stars
/reviews/*  v1  No stars

Most important part here is URI rewrite:-
Refer unit 1-GE bookshow, Unit3-GE-1st BookShow applications.

Backend "reviews" service only accepts "/reviews" endpoints, not versioned paths like "/api/v3/reviews", so OSSM rewrite the URI before forwarding.

Example:-
 /api/v2/reviews/1 -> /reviews/1 

Why rewrite is necessry?
If you route "/api/v2/reviews/1" directly to backend that only know "/reviews/1", backend return 404 error.
