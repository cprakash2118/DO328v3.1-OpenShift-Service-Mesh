1-# Traffic Routing (Basic implementation):-
Target:- Route all traffic to v1 

Step-1-# DestinationRule
$ oc explain ds |less
$ oc explain ds.spec |less
$ oc explain ds.spec.host |less

$ oc explain vs |less 
$ oc explain vs.spec |less 


vim destinationRule-file.yaml 
apiVersion: networking.istio.io/v1  
kind: DestinationRule
metadata:
  name: product-dr 
spec:
  host: product-service
  subsets:
  - name: v1 
    labels:
      version: v1 
  - name: v2
    labels:
      version: v2
	  
Step-2-# VirtualService

vim virtualService-file.yaml 
apiVersion: networking.istio.io/v1  
kind: VirtualService 
metadata:
  name: product-vs 
spec:
  hosts:
  - product-service
  http:
  - route:
    - destination:
	    host: product-service
	    subset: v1 

Outcome -> All traffic toward v1

3-# Header-Based Routing (More adv control)
Target -> Send only internal users to v2

Step-1-# DestinationRule
Step-2-# VirtualService

vim virtualService-file.yaml 
apiVersion: networking.istio.io/v1  
kind: VirtualService 
metadata:
  name: product-vs 
spec:
  hosts:
  - product-service
  http:                   ###### ------- From here 
  - match:
    - headers:
	    user-type:
		  exact: internal     # Example http://ip/internal 
    route:
	- destination:
	  host: product-service
	  subset: v2 
	  
  - route:
    - destination:
	    host: product-service
		subset: v1 
		
Outcome:-
- internal team -> v2 
- customers -> v1 

Test new feature internally before public release.

  name: product-vs 
spec:
  hosts:
  - product-service
  http:                   ###### ------- From here 
  - fault:
      delay:
	    percentage: 
		   value: 50 
		fixedDelay: 5s 
    route:
	- destination:
	  host: product-service
	  subset: v1 
	  
Result (Outcome):-
- 50% requests delayed by 5 secounds 

Why like this ?
- Test timeout handling 
- Validate resilience 
- Required for prod readiness

5-# Resilience (Retry, Timeout, Circuit Breaker).
Target -> Prevent cascading failure.

Retry + timeout###

Step-1-# DestinationRule
Step-2-# VirtualService

vim virtualService-file.yaml 
apiVersion: networking.istio.io/v1  
kind: VirtualService 
metadata:
  name: product-vs 
spec:
  hosts:
  - product-service
  http:                   ###### ------- From here 
  - route 
    - destination:
	    host: product-service
	retries:
	  attempts: 3 
	  perTryTimeout: 2s   ### A time-out value for each retry attempt. 
	timeout: 5s           ### Configures a 5-second timeout for all requests to the api-service.
	
Result:-
 - Retry failed calls automatically.
 - Avoid user-facing errors.
 
####- Circuit Breaker (DestinationRule)

Step-1-# DestinationRule
$ oc explain ds |less
$ oc explain ds.spec |less
$ oc explain ds.spec.host |less

$ oc explain vs |less 
$ oc explain vs.spec |less 


vim destinationRule-file.yaml 
apiVersion: networking.istio.io/v1  
kind: DestinationRule
metadata:
  name: product-dr 
spec:
  host: product-service
  trafficPolicy:
    connectionPool:
     tcp:                       ## TCP connection pool settings.
       maxConnections: 100      ## Maximum number of simultaneous TCP connections per replica.
       connectTimeout: 30ms     ## Maximum time to establish a TCP connection.
     http:                           ## HTTP connection pool settings.
       http1MaxPendingRequests: 50  ## Maximum number of HTTP requests pending per replica.
       maxRequestsPerConnection: 1  ## Maximum number of requests permitted on a single connection
	   
	   
spec:
  host: product-service
  trafficPolicy:
   outlierDetection:
     consecutive5xxErrors: 5    ### Number of consecutive 5xx errors before evicting the replica.
     interval: 10s              ### Time interval between checking error counts.
     baseEjectionTime: 30s      ### Minimum duration of replica ejection time.
     maxEjectionPercent: 50     ### Maximum percentage of evicted replicas for a service at any time.
	 
	 
Outcome:-
 - Bad pods automatically isolated 
 - System stays table 

Critical for:-
 - DB failure
 - External API failure
 - VM migration backed stability

6-# Traffic Mirroring:-
  Prod is on v1 (current and stable version)
  Test v2 (new version) without impacting users.
  
Step-1-# DestinationRule
Step-2-# VirtualService

vim virtualService-file.yaml 
apiVersion: networking.istio.io/v1  
kind: VirtualService 
metadata:
  name: product-vs 
spec:
  hosts:
  - product-service
  http:                   ###### ------- From here 
  - route:
    - destination:
        host: product-service
        subset: v1 
    - mirror:
        host: product-service
        subset: v2 

Result:-
 - User -> only see v1 response 
   v2 gets same traffic silently 

 - Test new  backend logice safely
 - compare performance
